Incident + Hotfix — 4.0.20 build 54 (2026-06-02)

Symptom: the distributed 4.0.19/53 build crashed on launch into the in-app ErrorBoundary ("Something went wrong — Maximum update depth exceeded") for users with iOS Reduce Motion enabled (user-reported screenshot).

Root cause (by elimination): 4.0.19 changed AccessibilityContext.isReduceMotionEnabled from the constant config.reduceMotion to systemReduceMotionEnabled || config.reduceMotion, where systemReduceMotionEnabled is seeded by an async AccessibilityInfo.isReduceMotionEnabled() probe that resolves after mount. For a Reduce-Motion user this flips shouldReduceMotion false → true at runtime, re-triggering motion-transition effects across animated screens and driving an infinite render loop. Build 53 was the first distributed build to expose it because earlier builds kept shouldReduceMotion constant. Exhaustive static review of every reduce-motion consumer found no single-file setState loop, which is consistent with a transition-triggered interaction rather than one bad component.

Fix

  • AccessibilityContext: isReduceMotionEnabled = config.reduceMotion (stable for the whole session); removed the systemReduceMotionEnabled state, its async probe, and the reduceMotionChanged subscription.
  • Removed the mirror-state anti-pattern: config is derived directly from settings.accessibility (single source of truth) instead of being copied into local state via an effect; updateConfig persists through settings.
  • Added __tests__/accessibility.reduceMotion.behavior.spec.tsx — a behavioral render test (mounts the provider). The previous reduce-motion "tests" only string-matched source, which is why the regression shipped green.

Validation evidence

  • New behavioral test fails on the pre-fix code (2/3 assertions) and passes on the fix — reproduces and guards the trigger.
  • bun run test — 108 suites passed (1 skipped), 881 tests passed (13 skipped), 0 failures. tsc 0 errors. lint exit 0. React Doctor 100/100 both projects.
  • On-device-class verification: built Release for the iOS Simulator (via xcodebuild; the expo run:ios path was blocked by an unrelated expo-cli DOMParser error) and launched it on a Simulator with system Reduce Motion enabled — the fixed build renders onboarding normally with no ErrorBoundary, i.e. the original crash condition no longer crashes.
  • Signed Apple Distribution archive + export + asc publish testflight → processingState VALID; asc validate testflight --strict 0 errors / 0 warnings; bilingual notes set; distributed to the internal group. Fixed build also reinstalled on the paired physical device.

Residual risk / deferred

  • An in-app Reduce-Motion toggle still transitions shouldReduceMotion at runtime; static review found no consumer-side setState loop, but this path was not exercised through the UI (Release build exposes a flat a11y tree; deep navigation not drivable here). Recommend confirming the toggle on the installed device build.
  • OS Reduce-Motion auto-detection is intentionally removed. A safe re-introduction (e.g. a value that is correct from first paint, avoiding a post-mount transition) is deferred until it can be verified on-device with Reduce Motion enabled.
