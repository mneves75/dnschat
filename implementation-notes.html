Design Decisions

Decision Reasoning Status
Keep DNS privacy warnings explicit in UX/docs. The product contract says DNS is observable infrastructure; UI and docs must not imply DNS prompts are private. Accepted
Prefer focused fixes over broad redesign. The app already has a release-tested design system and broad prior verification; changes should target concrete production risks found in this pass. Accepted
Do not add dependencies unless a current blocker proves they are necessary. Repo instructions prefer deletion, existing utilities, and local patterns; no dependency gap has been proven yet. Accepted
Use per-transport DNS probes in onboarding demo instead of the normal fallback query. The old demo could report Native DNS success even when queryLLM succeeded through a fallback. Forced testTransport probes keep the UI claim aligned with the actual transport being shown. Accepted
Fix iOS resolver copy drift by synchronizing the tracked prebuild copy and extending the sync gate. The built module source lives in modules/dns-native/ios/, but ios/DNSNative/ remains tracked. A sync check is cheaper and safer than silently leaving stale native code in the repo. Accepted
Deploy to TestFlight at the end of the run if release credentials and signing are available. The user explicitly requested TestFlight after the initial review goal. This makes build-number sync, archive/export, upload, and ASC validation part of the final release lane. Accepted

Tradeoffs

  • Use prior memory as context only. Re-run current checks before claiming the current implementation is clean.
  • Use browser/UI smoke after frontend-visible edits. If no UI code changes are needed, browser smoke still gives confidence but is not a substitute for native release gates.
  • Keep network-setup onboarding as a configuration animation rather than live network probing. The DNS demo now probes transports directly; the setup screen labels only the fallback order being configured.
